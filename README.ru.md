# Pi Session Restore

Расширение VS Code для восстановления сессий Pi во встроенном терминале. Восстановление работает по мере возможности: расширение не оживляет закрытый процесс, а помогает заново открыть последнюю связанную с терминалом сессию.

## Как работает

- При активации расширение добавляет `resources/bin` в начало `PATH` только для новых встроенных терминалов VS Code через `ExtensionContext.environmentVariableCollection`.
- Дополнительно расширение устанавливает модуль Pi в `~/.pi/agent/extensions/pi-vscode-session-restore`. Он сообщает достоверные события `session_start` после `/resume`, `/new`, `/fork` и запуска Pi из командной строки с восстановлением сессии.
- `resources/bin/pi` безопасно находит настоящий исполняемый файл `pi` дальше по `PATH`, исключая собственную директорию.
- При обычном интерактивном запуске обёртка заранее назначает JSONL-файл сессии Pi и запускает настоящую команду как `pi --session <path> ...`. Команды с явно заданной сессией, восстановлением, печатью или управлением пакетами не переписываются.
- `/resume` и `--resume` не получают новый `--session`: выбранный файл сессии Pi записывает в общий журнал событий модуль со стороны Pi.
- Псевдонимы вроде `p='pi'` расширение не создаёт, но они продолжают работать: оболочка раскрывает псевдоним в `pi`, после чего используется обёртка из `PATH`.
- Учёт на стороне VS Code сохраняет локальные записи восстановления по событиям обёртки, оболочки и модуля Pi. Интеграция VS Code с оболочкой используется как дополнительный источник, если она доступна.
- Команда восстановления строится через проверенный локально синтаксис `pi --session <path|id>`.
- Записи восстановления хранят снимок названия вкладки терминала (`terminalName`) и используют его при создании новых терминалов для восстановления. Публичный API VS Code не позволяет переименовывать уже существующие терминалы, запущенные через оболочку.
- После `/resume`, `/new`, `/fork` и других переключений сессии Pi активной для вкладки терминала считается последняя `sessionPath`; предыдущая сессия помечается неактивной через `terminalClosedAt`.
- Если пользователь закрывает вкладку терминала с сессией Pi, запись получает `terminalClosedAt`. При автоматическом восстановлении после запуска такая запись используется только если VS Code действительно восстановил вкладку терминала с тем же названием. Это защищает от ложных событий закрытия при закрытии всего окна.

## Команды

- `Pi Session Restore: Show Records`
- `Pi Session Restore: Clear Records`
- `Pi Session Restore: Restore Last Session`

## Настройки

- `piSessionRestore.enabled`: включает обёртку через `PATH` для новых терминалов.
- `piSessionRestore.sessionGlobPaths`: шаблоны путей к JSONL-файлам сессий; по умолчанию `~/.pi/agent/sessions/**/*.jsonl`.
- `piSessionRestore.restorePolicy`: `off`, `prompt`, `auto-confident`; по умолчанию `auto-confident`.
- `piSessionRestore.confidenceThreshold`: `high`, `medium`, `low`.
- `piSessionRestore.diagnosticsLevel`: `off`, `error`, `info`, `debug`.
- `piSessionRestore.recordTtlDays`: срок хранения локальных записей.
- `piSessionRestore.installPiExtension`: устанавливает модуль Pi для точного отслеживания `/resume`; по умолчанию включено.

## Приватность

Расширение хранит только локальные диагностические события от обёртки и модуля Pi, а также записи восстановления в `globalStorageUri` VS Code. Содержимое сессий не копируется. При `piSessionRestore.diagnosticsLevel=debug` канал вывода `Pi Session Restore` дополнительно показывает область рабочей папки, найденные терминалы, кандидатов на автоматическое восстановление и причины пропуска.

## Ограничения

- Не восстанавливает живой процесс терминала.
- Не поддерживает tmux, Remote SSH, WSL и Dev Containers.
- Автоматическое восстановление намеренно консервативно: оно работает только при `restorePolicy=auto-confident`, записях с высокой уверенностью и известной рабочей папке. Если VS Code восстановил несколько неактивных вкладок терминала в одной рабочей папке, расширение пытается восстановить соответствующее число последних записей Pi для этой папки.
- Автоматическое восстановление использует только уже существующие неактивные вкладки терминала, восстановленные самим VS Code. Расширение не создаёт новые вкладки при запуске, чтобы не спорить с собственным механизмом восстановления терминалов VS Code и не создавать дубли. Если у восстановленных вкладок есть сохранённые названия, записи сначала сопоставляются с вкладками по названию и только потом, как запасной вариант, по времени.
- Название вкладки терминала сохраняется по мере возможности: API VS Code позволяет читать `Terminal.name`, но не даёт публичного способа переименовать уже существующий терминал, запущенный через оболочку.
- Интеграция VS Code с оболочкой может быть недоступна. В этом случае расширение использует журнал событий обёртки и модуля Pi, а для ручного восстановления отправляет команду через `terminal.sendText`.
